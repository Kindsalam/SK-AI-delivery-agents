Delivery team agents

Twenty AI agents that help delivery teams run cadence at three tiers. The team running Scrum. The Scrum of Scrums where two to seven teams align. The leadership cadence where the unit of work is the quarter. Plus a substrate (transparency-scoreboard, which is a dashboard, not an agent) and one I deprecated (health-check).

Each agent prepares the room for a real conversation. The agent drafts. The product owner sharpens. The team commits. The leadership table decides. Final call always sits with humans.

Agent Tier Cadence What it does How to set it up Data hygiene needed
Tier 1. Team Scrum cadence (single team, sprint-aligned)
standup-update-agent Team Daily, before 9am Drafts the overnight one-pager so the team walks into the Daily Scrum already informed. MCP servers for Jira + GitHub + Slack/Teams; schedule daily 6–8am team-time. Jira ticket states reflect reality (no fortnight-stale tickets); commits reference ticket IDs; Sprint Goal written in Jira sprint metadata or one canonical Confluence page.
pragmatic-brutal-facts-agent Team Daily, 6:30am Flags drift against the team's own baseline so problems surface days, not sprints, late. MCP for Jira + GitHub + bug tracker; precompute the 60–90 day baseline; schedule daily 6:30am. 60–90 days of sprint history in Jira; bug states reflect reality (no "fixed" tickets that aren't); sprint commitments tracked in Jira, not just on a board.
refinement-prep-agent Team Epic-signoff + morning of refinement Drafts first-pass Jira stories under an Epic so the PO edits and the team discusses, not types. MCP for Jira (read + write to Draft) + Confluence; trigger on Epic-signoff and morning of refinement; shadow mode for first week. Epics have description + Epic-level acceptance criteria; Done stories share a consistent shape (the pattern library); team's Definition of Ready agreed and written down; design / spec pages linked on Epics.
sprint-planning-prep-agent Team Morning of Planning + 24h draft Drafts Sprint Goal candidates with a capacity-checked Sprint Backlog per option so the room commits, not authors. MCP for Jira + Confluence + calendar; schedule morning of Sprint Planning and 24h before; never writes to Jira sprint metadata. Parent Initiative has a named top metric for the quarter; six sprints of velocity history; calendar / leave / on-call accessible; past Sprint Goals recorded in one canonical place.
retro-prep-agent Team Morning of retro Drafts candidate themes and candidate actions so the team decides what to try, not fishes for ideas. MCP for Jira + Confluence + brutal-facts feed; schedule morning of retro; shadow mode first week. Retros land in one canonical place every sprint; action items tracked with completion status; team's retro format agreed (Start/Stop/Continue, 4Ls, etc.).
retro-synthesis-agent Team End of every sprint; full pass quarterly Reads every retro across a quarter and surfaces themes that recur versus themes that got fixed. MCP for the Confluence retro archive (or whichever retro tool the team uses); schedule end of every sprint, full pass at quarter-end. 4–12 sprints of retro history in one canonical place; themes recorded consistently across retros; action-item completion tracked.
sprint-goal-tracker-agent Team Daily Watches the Sprint Goal trajectory so it is not written at planning and forgotten until review. MCP for Jira sprint metadata; schedule daily; trajectory line is injected into the standup-update one-pager. Sprint Goal written in Jira sprint metadata (or one canonical Confluence page); items linked to the goal via label, fix-version, or custom field.
team-dora-agent Team Daily snapshot, weekly card Posts the four DORA numbers with trend arrows; pins to the transparency scoreboard. MCP for CI/CD + incident tool + GitHub (joined for lead-time-to-prod); schedule daily, summarise weekly. Production deploys uniformly tagged in CI/CD (one workflow, one tag); incidents linked to the deploy that caused them; PR-to-deploy traceability across the pipeline.
sprint-review-prep-agent Team · also-considered Morning of Review + 24h draft Drafts the review pack so the PO is not scrambling on the morning to remember what shipped. MCP for Jira + Confluence + GitHub; schedule 24h before and morning of Review. Done items have AC populated; PRs and commits linked to tickets; demo-ready flag (or convention) used consistently if the team uses one.
Tier 2. Scrum of Scrums and multi-team coordination (program + portfolio scale)
cross-team-dependency-agent Multi-team Daily, before SoS Reads cross-team dependency links so the SoS does not open with twenty minutes of cross-team status. MCP for Jira (across all participating teams' boards) + Confluence + Slack/Teams; schedule daily, 30 min before SoS. Teams use Jira is blocked by / blocks consistently; cross-team Epic links populated; component ownership documented in Confluence.
integration-readiness-agent Multi-team Daily during integration week Drafts an honest go / no-go plus proposed unblock actions before the integration meeting walks in. MCP for GitHub + CI/CD + feature-flag service + Jira; schedule daily during integration week, continuously the day of. Integration tests exist and run in CI; contract tests run on cross-team APIs; feature flags tracked in one known place; items tagged for the integration in Jira.
multi-team-sprint-rollup-agent Multi-team End of every sprint Stitches each team's sprint-end signal into a program-level read for the program manager / RTE. Reads outputs of team-level agents (sprint-goal-tracker, standup-update) plus Jira at Initiative level; schedule the morning after each team's review. Every team in scope runs sprint-goal-tracker (so the goal-verdict signal exists); Initiative-level Jira links populated across teams; multi-team OKRs in one known place if used.
program-risk-radar-agent Multi-team Daily summary, weekly deep Lifts risk to where it compounds: across teams, dependencies, integrations, milestones. RAID-style. MCP for Jira RAID + outputs of cross-team-dependency and integration-readiness; schedule daily summary, weekly deep pass. RAID logs maintained per team or per program; risks have probability + impact scores and named mitigation owners; deadlines and milestone dates tracked in the program plan.
portfolio-sprint-rollup-agent Portfolio End of every sprint Stitches each program's rollup into a portfolio-level read for the portfolio leader. Reads outputs of every program's multi-team-sprint-rollup; schedule end of every sprint at portfolio scale; shadow mode for two sprints. Every program in scope runs multi-team-sprint-rollup cleanly; strategic-bets register exists with hypothesis + spend per bet; portfolio-level OKRs and milestones tracked in one canonical place.
chapter-throughput-agent Multi-team · also-considered Weekly Surfaces the engineering chapter's pattern across teams: who drags the average, what cross-team patterns show. Reads team-dora outputs across all teams in the chapter + Jira throughput metrics; schedule weekly. team-dora-agent running cleanly on every team in the chapter for at least one quarter; DORA data accessible across teams under a single chapter view.
Tier 3. The LT cadence (quarter-aligned, leadership audience)
qbp-pack-prep-agent LT First week of new quarter Drafts the QBP pack so the LT walks in to candidate objectives, KRs, and bets to react to. MCP for OKR sheet + Jira + Confluence + opportunity-tree + win/loss feed; schedule first week of new quarter; never updates the OKR sheet directly. QBR pack from the just-completed quarter exists; OKR scorecard with weekly confidence ratings; strategic-bets register populated with hypothesis + evidence + spend per bet; opportunity-tree maintained.
qbr-pack-prep-agent LT Twice-weekly in final two weeks of quarter Drafts the QBR pack so the LT walks in to a pack everyone has read, not slides nobody finished. MCP for north-star metric warehouse + OKR sheet + Jira + product-analytics + win/loss; schedule twice-weekly in final two weeks of quarter. North-star metric tracked and named; OKR scorecard with weekly confidence ratings; strategic-bets register; last quarter's QBR pack archived for delta tracking.
okr-checkin-agent LT Monday 7am, Friday 4pm Runs the Wodtke weekly check-in; confidence rating per KR plotted against last week's number. MCP for the OKR sheet (Confluence, Lattice, Ally.io, etc.) + product-analytics + Jira; schedule Monday 7am and Friday 4pm. One canonical OKR location (not three competing ones); each KR has a confidence column; product-analytics events tied to each KR named in the sheet; Jira initiatives linked to KRs.
roadmap-status-agent LT Weekly digest, monthly delta Keeps the roadmap honest with what has happened, not what was remembered before the monthly meeting. MCP for Jira + ProductBoard / Aha (or equivalent) + delivery capacity feed; schedule weekly digest, monthly deep document. One canonical roadmap location; Jira initiatives have target quarters / milestones attached; last month's roadmap snapshot saved somewhere stable for delta comparison.
opportunity-tree-maintenance-agent LT Friday afternoon Updates Torres's opportunity-solution tree with the week's discovery signals as candidates the PM merges. MCP for discovery transcripts (Notion native; Dovetail and Otter via custom wrappers) + product feedback + the canonical tree; schedule Friday afternoon. Discovery interviews are transcribed each week; the opportunity-solution tree exists as a canonical artefact (not just slides); customer-feedback channels tagged consistently.
win-loss-synthesis-agent LT Monthly digest + quarterly synthesis Turns the win/loss programme that exists on paper into a real synthesis the LT actually reads. MCP for CRM + sales-call transcripts (where consented) + product-analytics + churn surveys; schedule monthly digest + quarterly synthesis. A real win/loss programme exists (interview discipline, not just CRM closed-reason fields); CRM closed-won and closed-lost reasons structured and tagged consistently; sales-call transcripts saved where consented.
portfolio-bets-readout-agent LT Monthly, five business days before steering Prepares the steering / board readout so the head of product is not writing the same paragraphs every month. MCP for the strategic-bets register + OKR scorecard + win/loss feed; schedule monthly, five business days before steering. Strategic-bets register has hypothesis + evidence + spend per bet; north-star metric tracked and named; LT has agreed thresholds for each colour-coded signal (red / amber / green) before the agent runs.
design-partner-state-agent LT · also-considered Monthly Tracks each design partner's engagement state so the drifting one does not show up on a churn list. MCP for design-partner CRM + product-analytics + support tickets; schedule monthly. Assumes a real design-partner programme exists. Design partners tracked in a CRM with a canonical partner ID (not split across systems); per-partner usage data accessible; convert-to-paying milestones recorded; scheduled-call cadence tracked.
strategic-account-product-review-agent LT · also-considered Quarterly Reviews the top-ten enterprise accounts so the head of product is not reading ten Slack channels on a weekend. MCP for CRM + customer-success notes + usage data; schedule second-to-last week of quarter. Assumes enterprise CS maturity. Account usage data tracked per account; customer-success notes written in one canonical place; expansion-revenue tracking; churn-risk score available; feature-request log per account.
Substrate and deprecated
transparency-scoreboard Substrate · not an agent Live dashboard Read-only dashboard holding the team's real numbers in one place everyone can see. Build in your existing dashboard tool (Tableau, Looker, Metabase, Notion). Name a metric-definition owner outside the chain of command. Metric definitions agreed and written down where the team can see them; data sources connected and refreshed reliably; metric-definition owner sits outside the chain of command of whoever the metrics judge.
health-check-agent Deprecated n/a Reading chat tone and 1-on-1 notes for burnout signals carries severe surveillance risk. Do not build. The deprecation rationale is in the article and in agents/deprecated/health-check.md. N/A. Burnout signals are picked up better by humans paying attention; make sure your retros and 1-on-1s have space for the signal to come up.
